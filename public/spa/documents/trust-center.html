§1

脅威モデル & 信頼境界

暗号システムを評価する前に、脅威モデルを明確にする必要があります。ProofMarkは特定の、限定された主張に対処します。

唯一の検証可能な主張

Claim: SHA-256ハッシュ H を持つファイルが、信頼された第三者TSAによって証明されたタイムスタンプ T の時点で、改ざんされていない状態で存在していた。

✓ In Scope（証明できること）

  • ✓ 時刻Tにおけるファイル完全性
  • ✓ タイムスタンプの否認不可性
  • ✓ 第三者による独立検証
  • ✓ 発行後はProofMark依存なし

✗ Out of Scope（証明できないこと）

  • ✗ 著作権の帰属
  • ✗ 芸術的独自性
  • ✗ 世界初の創作であること
  • ✗ AI生成コンテンツの合法性

Adversary Model

ProofMarkが対処する攻撃者と防御策
攻撃者 能力 ProofMarkの防御
主張偽造 先に作ったと主張 TSAタイムスタンプが事実を証明；ハッシュ衝突は不可能（2¹²⁸）
ファイル改ざん タイムスタンプ後に改ざん SHA-256ダイジェストが即座に無効化；第三者が検証可能
タイムスタンプ再利用 古いTSTを新ファイルに TSTはhash+nonceをバインド；暗号的に検知可能
サーバー侵害 DBハッシュ漏洩 ハッシュは公開安全；Private Proofモードでは原画なし
TSA共謀 偽タイムスタンプ発行 TSA選定（§4）+ RFC3161チェーン検証で軽減
§2

暗号ハッシュ — SHA-256 via Web Crypto API

ProofMarkはブラウザネイティブの Web Crypto API（W3C仕様）を使用し、SHA-256ダイジェストを完全にクライアントサイドで計算します。Private Proofモードでは原画はサーバーに一切送信されません。

実装コード

TypeScript 
// ProofMark client-side hashing pipeline
// Requires: SubtleCrypto (window.crypto.subtle)

async function computeProofHash(file: File): Promise {
// 1. 非対応ブラウザを明示的に拒否 — フォールバックなし
if (!window.crypto?.subtle) {
throw new Error(“SubtleCrypto unavailable — use a modern browser”);
}
// 2. ArrayBufferとして読み込み（ブラウザメモリ内で完結）
const buffer = await file.arrayBuffer();
// 3. SHA-256計算 — ハードウェアアクセラレーション
const hashBuffer = await window.crypto.subtle.digest(“SHA-256”, buffer);
// 4. 小文字hex文字列（64文字）にエンコード
return Array.from(new Uint8Array(hashBuffer))
.map(b => b.toString(16).padStart(2, “0”)).join(””);
}
⚠ No-Fallback Policy. SubtleCryptoが利用できない場合（非HTTPS、レガシーブラウザ）、UIは明示的なエラーを表示します。JSポリフィルへのサイレントフォールバックは行いません。

SHA-256のセキュリティ特性

SHA-256アルゴリズムのセキュリティ特性一覧
特性 値 / 保証
出力サイズ 256ビット（32バイト）— 64文字hex
衝突耐性 2¹²⁸回演算（誕生日攻撃の下界）
原像耐性 2²⁵⁶回演算 — 計算的に不可能
雪崩効果 1ビット変化 → 出力の約50%が変化
標準 NIST FIPS 180-4 — TLS 1.3、Git、Bitcoinでも使用
§3

RFC3161 タイムスタンプ — アーキテクチャ & 信頼チェーン

RFC3161は、ハッシュ値を特定の時刻にバインドする暗号的に安全なプロトコルを定義します。生成されたTime-Stamp Token（TST）は、ProofMarkを信頼することなく独立して検証可能です。

プロトコルフロー

  1. ①BrowserSHA-256(file) を計算 → ダイジェストH（32バイト）
  2. ②Browser暗号ノンスN生成（リプレイ攻撃防止）
  3. ③BrowserTimeStampReq := { hashAlgorithm: SHA-256, hash: H, nonce: N, certReq: true }
  4. ④→ TSATSAへHTTPS POST — ハッシュのみ送信、ファイルは送信しない
  5. ⑤TSA{ hash, time, cert } を秘密鍵で署名 → TST生成
  6. ⑥→ DBTST（DER/base64）をSupabaseに保存
  7. ⑦第三者TSA公開鍵でTSTを検証 — ProofMark不要

Time-Stamp Token（ASN.1構造）

ASN.1 
TimeStampToken ::= ContentInfo {
  contentType : id-signedData,
  content : SignedData {
    encapContent : TSTInfo {
      messageImprint : { SHA-256, hash_of_your_file },
      genTime        : GeneralizedTime,  -- e.g. 20260410153042Z
      nonce          : INTEGER,           -- replay protection
    },
    signerInfo : { TSA_cert_chain, RSA_signature }
  }
}

データ永続化 & セキュリティ（Supabase / RLS）

ProofMarkはSupabase（PostgreSQLベースのBaaS）を使用します。セキュリティはRow-Level Security（RLS）ポリシーによりデータベース層で強制されます。

スキーマ

PostgreSQL 
-- 証明レコード（追記専用）
CREATE TABLE proof_records (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      UUID NOT NULL REFERENCES auth.users(id),
  sha256_hash  CHAR(64) NOT NULL,
  proof_mode   TEXT CHECK (proof_mode IN ('private', 'shareable')),
  tst_base64   TEXT,        -- 完全なRFC3161 TST（DER/base64）
  tst_time     TIMESTAMPTZ, -- TSTから抽出したgenTime
  created_at   TIMESTAMPTZ DEFAULT now()
);

Row-Level Security ポリシー

PostgreSQL RLS 
ALTER TABLE proof_records ENABLE ROW LEVEL SECURITY;

– SELECT: 自分のレコードのみ
CREATE POLICY “own_records_select” ON proof_records
FOR SELECT USING (auth.uid() = user_id);

– INSERT: 自分のレコードのみ
CREATE POLICY “own_records_insert” ON proof_records
FOR INSERT WITH CHECK (auth.uid() = user_id);

– UPDATEとDELETEのポリシーなし → レコードは不変

– 公開検証用: shareableのみ
CREATE POLICY “public_shareable” ON proof_records
FOR SELECT USING (proof_mode = ‘shareable’);
設計上の重要な判断： (1) UPDATEポリシーなし — タイムスタンプ操作が不可能。(2) IPアドレスはSHA-256(IP)のみ保存。(3) tst_base64が完全なRFC3161 TSTを保持 — ProofMark停止後も外部検証が可能。
§6

エンドツーエンド データフロー

Private Proofモード（デフォルト）

  1. 1Browserファイル選択 — ブラウザメモリ内に留まる（FileReader API）
  2. 2BrowserSubtleCrypto.digest()でSHA-256計算 — ネットワーク通信ゼロ
  3. 3→ SupabasePOST /functions/v1/process-hash { hash, filename, size, nonce }
  4. 4Edge FnJWT検証（Supabase Auth）、user_id抽出
  5. 5→ TSARFC3161 TimeStampReqをTSAへHTTPS送信
  6. 6TSA →署名済みTSTを含むTimeStampRespを受信
  7. 7→ DBINSERT into proof_records { user_id, hash, tst_base64, tst_time }
  8. 8→ Browser{ proof_id, tst_time, certificate_url } を返却
  9. 9BrowserPDF証明書をクライアントサイドで生成 — 原画は一切送信されていない
プライバシー保証： Private Proofモードでは、原画ファイルはブラウザの外に出ません。SHA-256は一方向関数であり、ハッシュ値から元ファイルを復元することは計算的に不可能です。

検証ガイド — 第三者監査

ProofMarkの証明は、標準的なオープンソースツールを使って誰でも検証できます。ProofMarkインフラへの信頼は不要です。

Step 1 — SHA-256ハッシュの再計算

Python 3 
# Python 3 -- SHA-256ハッシュ検証
import hashlib

def verify_hash(filepath: str, expected: str) -> bool:
with open(filepath, “rb”) as f:
computed = hashlib.sha256(f.read()).hexdigest()
ok = computed == expected.lower()
print(“Result: PASS ✓” if ok else “Result: FAIL ✗”)
return ok

Step 2 — RFC3161タイムスタンプの検証

Shell (OpenSSL) 
# TSTを抽出（base64）
echo "<tst_base64>" | base64 -d > proof.tsr

# OpenSSLで検証 — ProofMarkへの依存ゼロ
openssl ts -verify \
  -in proof.tsr \
  -digest <sha256_hash_hex> \
  -CAfile freetsa-ca.crt \
  -untrusted freetsa-tsa.crt

# 期待される出力:
Verification: OK
```
TSAのルート証明書はTSAのウェブサイトで公開されています。この検証はProofMarkへの信頼を一切必要とせず、TSAの公開鍵とOpenSSLのRFC3161実装のみに依存します。
```

制限事項 & 正直な開示

制限事項についての誠実な開示が、それを隠すよりも信頼を築くと考えています。

  • ⚠
    Beta TSA（FreeTSA.org）SLAなし、自己署名ルートCA。有料リリース前に商用TSAへ移行予定。既存TSTは暗号的に有効のまま維持。
  • ⚠
    HSMなしTSA署名鍵はTSA運営者が管理。軽減策：本番では信頼できる商用TSAを使用。
  • ⚠
    証明ごとに単一TSAマルチTSAクロスタイムスタンプ（耐性向上）はProロードマップ。
  • ⚠
    ブラウザ信頼境界悪意のある拡張機能がハッシュ前のデータをインターセプト可能。全ブラウザ暗号ツールと共有するリスク。
  • ⚠
    C2PA未対応（現時点）C2PAマニフェストの埋め込みは未対応。長期ロードマップに含まれています。
  • ⚠
    第三者コード監査なしハッシュ計算ロジックをGitHubでMITライセンス公開。§Aの検証スクリプトで独立検証が可能。

Appendix：Python検証スクリプト（完全版）

ProofMark証明書を検証するスタンドアロンスクリプト。Python 3.8+とOpenSSLが必要。完全版ソース: github.com/proofmark-jp/verify

Python 3 
#!/usr/bin/env python3
"""ProofMark RFC3161 Timestamp Verifier
   Usage: python verify_proofmark.py <file> <hash_hex> <tst_base64>
"""
import sys, hashlib, base64, subprocess
from pathlib import Path

def verify_file_hash(filepath: str, expected_hex: str) -> bool:
actual = hashlib.sha256(Path(filepath).read_bytes()).hexdigest()
ok = actual == expected_hex.lower()
print(f”[HASH] Expected : {expected_hex}”)
print(f”[HASH] Computed : {actual}”)
print(”[HASH] Result   : PASS ✓” if ok else ”[HASH] Result   : FAIL ✗”)
return ok

def verify_rfc3161_tst(tst_b64: str, hash_hex: str,
ca: str = “freetsa-ca.crt”,
tsa: str = “freetsa-tsa.crt”) -> bool:
tsr = Path(”/tmp/proof.tsr”)
tsr.write_bytes(base64.b64decode(tst_b64))
r = subprocess.run(
[“openssl”, “ts”, ”-verify”,
”-in”, str(tsr), ”-digest”, hash_hex,
”-CAfile”, ca, ”-untrusted”, tsa],
capture_output=True, text=True
)
ok = “Verification: OK” in r.stdout
print(f”[TST]  openssl : {r.stdout.strip()}”)
print(”[TST]  Result  : PASS ✓” if ok else ”[TST]  Result  : FAIL ✗”)
return ok

if **name** == ”**main**”:
filepath, hash_hex, tst_b64 = sys.argv[1], sys.argv[2], sys.argv[3]
h_ok = verify_file_hash(filepath, hash_hex)
t_ok = verify_rfc3161_tst(tst_b64, hash_hex)
sys.exit(0 if (h_ok and t_ok) else 1)
